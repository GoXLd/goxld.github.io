# Cloudflare Workers: перестать искать команду, начать читать исполнение

Привет всем!

8 мая моей проблемой было не отсутствие логов у моих Cloudflare Workers. У меня уже был `wrangler tail`. Проблемой был путь к ним: открыть модалку в дашборде, скопировать команду, перейти в терминал, ввести токен, запустить стрим, а затем пытаться связать строки текста с конкретным исполнением.

Такой процесс приемлем для исключительного вмешательства. Но он становится мучительным, как только вопрос прост: *почему этот worker пропустил свой цикл?* или *на каком этапе случился этот сбой?*

> Одной фразой: я заменил ярлык «Tail Logs» кнопкой `Debug` для каждого worker'а, **четырьмя уровнями** логов и `run_id`, который связывает начало, этапы, скипы, успех и ошибку одного исполнения.
{: .prompt-info }

## Отправная точка: логи доступны, но нет диагностического маршрута

| Зона | Исходное состояние |
|---|---|
| Дашборд | Действие `Tail Logs` копирует команду `wrangler` |
| Терминал | Перед запуском стрима нужно ввести токен |
| Worker | Уровни `silent`, `error` и `info` |
| Исполнение | Полезные сообщения, но без систематической корреляции между этапами |

Настоящий изъян — в разделении ответственности. Дашборд уже знал выбранный worker, его аккаунт и состояние. И всё же он отправлял оператора в shell ещё до первой полезной информации.

CLI-команда остаётся незаменимой в некоторых случаях — особенно живой tail с административной машины — но она больше не должна быть первым инструментом для понимания текущего состояния.

## Правило: debug должен быть временным и явным

Я сохранил очень простой принцип: уровень логов — это конфигурация деплоя, а не галочка, влияющая только на интерфейс. Включение debug передеплоивает worker с `LOG_LEVEL=debug`; выключение — с `LOG_LEVEL=error`.

>Правила, применённые к потоку:
>1. **Один worker — один переключатель** — действие никогда не касается неявной группы.
>2. **Уровень читается из bindings** — дашборд показывает реально сконфигурированное состояние, а не локальную настройку.
>3. **Debug остаётся временным** — он служит расследованию, затем возвращается к `error`.
>4. **Секреты не проходят через дашборд** — CLI-команда сохраняет тихий ввод токена, когда нужен терминальный tail.
{: .prompt-danger }

Со стороны API список Workers отдаёт `log_level`, `debug_enabled` и URL событий наблюдаемости. Дашборд может показывать `ON` или `OFF`, применять переключение и предлагать ссылку `Logs` только при активном debug.

```js
// Отображаемое состояние отражает реально задеплоенную конфигурацию.
function workerDebugEnabled(worker) {
  return worker?.debug_enabled === true
    || String(worker?.log_level || '').trim().toLowerCase() === 'debug';
}
```

## Четыре уровня вместо бинарного потока

Изменение в двух шаблонах Workers не сводится к печати большего количества текста повсюду. Оно вводит иерархию, позволяющую выбирать приемлемый уровень шума.

| Уровень `LOG_LEVEL` | Содержимое |
|---|---|
| `silent` | Никаких логов |
| `error` | Только ошибки |
| `info` | Основные этапы и сводка исполнения |
| `debug` | Дополнительные детали, полезные для расследования |

Уровень `debug` выше `info`: обязательные этапы остаются читаемыми без раскрытия деталей, а точечные детали появляются только во время поиска проблемы. Новые Workers провижинятся с `LOG_LEVEL=error`; mass redeploy сохраняет уже установленный уровень вместо его сброса.

## Исполнение становится читаемой последовательностью

Второе изменение важнее кнопки. Каждое исполнение получает идентификатор, создаваемый при старте. Оба шаблона затем пишут структурированные события вокруг этого идентификатора: `run_start`, `run_skip`, `run_stage`, `run_end` и `run_error`.

```js
const runId = createRunId();
ctx.logInfo(`[run_start] id=${runId} trigger=${ctx.trigger}`);

// Финальная строка связывает длительность и счётчики с тем же run_id.
ctx.logInfo(`[run_end] id=${runId} ok=true duration_ms=${durationMs}`);
```

Следствие операционное: `run_skip` больше не выглядит загадочным отсутствием. Он несёт причину — например `trigger_interval`, `cooldown`, `no_items` или `sleep_window`. Ошибка сохраняет тот же идентификатор и добавляет счётчики, доступные на момент сбоя. Для завершённого прогона сводка содержит, в частности, длительность и счётчики fetch и ответов `429`.

## Маршрут после изменения

| Шаг | Было | Стало |
|---|---|---|
| Выбрать worker | Открыть модалку и скопировать команду | Кликнуть `Debug` в его строке |
| Включить детали | Вручную запустить `wrangler tail` | Передеплоить этот worker с `LOG_LEVEL=debug` |
| Читать события | Терминальный стрим без единообразной корреляции | Ссылка `Logs` и события, связанные по `run_id` |
| Вернуться к тишине | Вручную менять деплой | Выключить `Debug`, редеплой в `error` |

CLI не исчез. Когда он нужен, документация сохраняет команду, запрашивающую токен через `read -s` — то есть без добавления в историю shell'а. Дашборд не заменяет административный доступ: он просто сокращает количество шагов до диагноза.

## Честная часть: debug — не бесплатная телеметрия

Ловушкой было бы оставить `debug` включённым по умолчанию. Детали полезны, когда ищешь конкретный переход — извлечённый список ссылок, этап парсинга, причину скипа — но они также увеличивают объём логов для чтения. Поэтому провижининг стартует с `error`, и поэтому кнопка `Debug` — явный выбор для каждого worker'а.

Другое ограничение: эта работа структурирует прикладные логи, но сама по себе не гарантирует, что причина инцидента находится в Worker'е. `run_error` может показать момент и счётчики, но внешняя зависимость или неверная конфигурация всё ещё требуют привычных проверок.

## Итоги

| Элемент | Результат |
|---|---|
| Уровни логов | **4**: `silent`, `error`, `info`, `debug` |
| Прицельность | Переключатель `Debug` на каждый worker |
| Корреляция | Один `run_id` на исполнение |
| Структурированные события | `run_start`, `run_skip`, `run_stage`, `run_end`, `run_error` |
| Возврат к номинальному уровню | Редеплой с `LOG_LEVEL=error` |

Прогресс не в том, чтобы иметь «больше логов». Он в том, чтобы уровень детализации соответствовал заданному вопросу, а полное исполнение читалось как одна история.

Для Workers, работающих регулярно, правильный вопрос больше не «как запустить tail?». А: *какое исполнение идёт сейчас, где оно остановилось и какой уровень информации нужно включить, чтобы это доказать?* Когда ответ начинается в дашборде и заканчивается `run_id`, диагностика становится заметно менее механической.

# ASN — автомагистраль скрапинга

## Как я сократил расходы на прокси и перестал воевать с 429

> <b>Предупреждение:</b> эта статья документирует техническое исследование и анализ рынка.
> Некоторые скриншоты могут содержать исторически чувствительные данные, но они больше не пригодны для эксплуатации в проде.
> Решение не замазывать их сознательное — чтобы поделиться реальным и проверяемым методом.
> Критические отзывы приветствуются.
{: .prompt-info }

В информатике многие проблемы упрощаются, когда находишь хорошую аналогию из реального мира. Для меня такой аналогией стало: автомагистраль = ASN.

![Webshare](img/asn/forfait_webshare.png){: .shadow }
*Видимый результат в счёте за прокси после оптимизации.*

> Цифра намеренно броская, но главное здесь — метод.
> Да, переход сразу на премиальные резидентные прокси (или другой сервис, например Bright Data) часто быстрее поднимает процент успеха[^residential].
> Но этот комфорт дорого стоит и маскирует реальную причину ошибок.
> Здесь я делюсь прежде всего выжимкой из трёх месяцев подстроек нескольких мировых пулов, чтобы держать производительность, не полагаясь только на бюджет.
> На практике многие решения «scraper-as-a-service» остаются дорогими и ненадёжными на специфических целях (например, Leboncoin или Steam Market) — отсюда интерес собирать цепочку под собственный сценарий.
{: .prompt-tip }

Решающим аргументом стала **детекция «заглушечных» ответов**.
Вы обходите Cloudflare, сервер отвечает `200`, вы получаете HTML... но это лишь пустая страница или фейковый контент.
Для сервиса *pay as you go* запрос тарифицируется как «успешный», хотя бизнес-результат нулевой.
Это как доставленный конверт без письма внутри.

![Пример пустого ответа при статусе 200](img/asn/small_reponse.png){: .shadow }

> Доставить ответ не значит доставить полезные данные.
{: .prompt-warning }

В этой статье я объясняю, как и почему это сработало.

## Как всё началось {#comment-tout-a-commence}

Про бизнес-контекст моих тестов есть отдельная статья:
[Почему Steam Market — отличный полигон для обучения]({% post_url ru/2026-01-14-pourquoi-steam-market %}).

Я работал почти непрерывно (трёхмесячный марафон без выходных) над собственным скрапером. Я начинал не с нуля: у меня уже были продовые скрипты Google Apps Script (`.gs`).
`gXd.node/scripts`
{: .filepath}
![Terminal Logs - local.node - scripts](img/asn/parsing_terminal.webp){: .shadow }
Но со временем процент успеха `urlFetch` мог падать до 15 % на некоторых целях, что становилось критичным.
Даже при высокой теоретической дневной квоте (часто называют около 20k запросов/день[^urlfetch]) реальные лимиты использования сложнее.
На практике в моём сценарии зона стабильности была около **10k запросов/день** с `urlFetch`, но порог остаётся динамическим:
при увеличении частоты ошибки порой появлялись и до этого объёма.

![Пример ограничения Apps Script](img/asn/appscript_limits.webp){: .shadow }

Поэтому я постепенно усложнял архитектуру.

Раньше всё было просто:

- добавить jitter,
- ограничить частоту,
- включить ротацию IP,
- сменить страну (в редких случаях)

Но в последние годы (особенно с подъёмом ИИ) бэкенды крупных сервисов сильно усложнились. Простые методы перестали работать.

Ошибки **429** начали появляться даже при осторожной нагрузке, в том числе со старыми методами обхода.

> При этом я никогда не превышал 1 запрос/секунду и не использовал агрессивный мультитрединг.
> В среднем каждый скрипт держался около 1 запроса/минуту к целевому сервису.
> Остальной трафик шёл по приоритетным каналам (внутренние API, база данных, системные сервисы), то есть вне классического скрапинг-потока.
{: .prompt-danger }

## Моя база защиты (усиленная анти-блокировочная версия) {#ma-base-de-defense}

К моменту тестов у меня уже была довольно развитая система:

- не более одного запроса в секунду на 100+ прокси,
- динамический тайминг + jitter 15 %
```js
function withJitter(baseDelayMs = 1000, jitterPercent = 0.15) {
  const delta = baseDelayMs * jitterPercent;
  return Math.round(baseDelayMs + (Math.random() * 2 - 1) * delta);
}
```
{: file="scripts/jitter.js"}

- автоматическая смена адресов
```js
class ProxyRotator {
  constructor(proxies) {
    this.proxies = proxies;
    this.index = 0;
  }

  next() {
    const proxy = this.proxies[this.index % this.proxies.length];
    this.index += 1;
    return proxy;
  }
}
```
{: file="scripts/proxy-rotator.js"}

- цепочка отката: Google → Proxy → Node
Логика намеренно прогрессивная:
сначала пробуем адреса Google (лучшая латентность/стоимость),
потом переходим на ротационные прокси,
и наконец на выделенные IPv4-инстансы (приватный пул, используемый только мной), когда цель становится слишком строгой.

![Процент успеха по цепочке AppScript -> Proxy -> Node](img/asn/success_rate_appscript_proxy_node.webp){: .shadow }
- собственные серверы в нескольких датацентрах с «чистыми» IPv4

Ноды сегодня распределены между Францией, Германией, Нидерландами и США.
Расширение на Сингапур, Швейцарию и новые точки в США запланировано, но пока не в приоритете.

![Текущее распределение нод](img/asn/nodes.png){: .right .shadow }

Текущая доступность держится в целом между **98 % и 99 %**, что позволяет приоритизировать программную оптимизацию перед географической экспансией.
![Доступность нод](img/asn/nodes_success_rate.png){:.left .shadow }

Логи на сервере gXd.node
`gXd.node/logs/scraper-traffic.log`
{: .filepath}
![Webshare](img/asn/logs.webp){: .shadow }
- система ротации «Светофор»,
- эмуляция разных устройств и браузеров.

Упрощённые выдержки логики (педагогическая версия):


```js
async function fetchWithFallback(request) {
  try {
    return await fetchViaGoogle(request);
  } catch {
    try {
      return await fetchViaProxy(request);
    } catch {
      return await fetchViaNode(request);
    }
  }
}
```
{: file="scripts/fetch-with-fallback.js"}

На бумаге всё было корректно.
Но результаты по-прежнему сильно варьировались по странам и по пулам.

## Странное наблюдение {#observation-etrange}

Со временем я заметил интересную закономерность.
В одной и той же стране разные прокси показывали:

- 90 % успеха,
- 70 %,
- иногда меньше 50 %.

Скриншот ниже подтверждает идею: при равной стране процент успеха сильно варьируется по ASN и по связанным блокам адресов.

![Различия успеха по ASN](img/asn/be_replaced_ips.webp){: .shadow }

Также видно, что у «слабого» ASN меньше запросов.
Это не искажение измерений — это прямой эффект оптимизации.

Как только прокси показывает дрейф (429, таймаут, нестабильная латентность), он быстро заменяется. Цель — не накопить 1000 отказов на мёртвом прокси, а отрезать потери как можно раньше.

Порог замены остаётся сознательно индивидуальным: он зависит от стоимости пула, толерантности к ошибкам и темпа сбора.

<b>При этом:</b>
- нагрузка была одинаковой,
- поведение одинаковым,
- тайминг одинаковым.

Разница сводилась к одному параметру: **ASN**.

> Серверы остаются онлайн, но качество прохождения может резко деградировать.
> Значит, цель — выбирать наименее загруженные «ASN-магистрали», чтобы держать трафик стабильным и рентабельным.
{: .prompt-tip }

![Выбирать наименее загруженные ASN-магистрали](img/asn/traffic_scrapper.png){: .shadow }

## Гипотеза: лимиты не по IP, а по ASN {#hypothese-asn}

<b>Гипотеза напросилась сама:</b>

<i>Современные сервисы всё чаще ограничивают не отдельные IP, а блоки адресов **целых автономных систем (ASN)**.</i>

Логика проста:
Если в блоке ASN много подозрительной активности, проще замедлить весь блок, чем фильтровать адрес за адресом.

Последствия:
- даже «чистые» IP начинают получать 429,
- процент успеха падает,
- расходы растут.

Если в том же блоке ASN (/24 /23 /22 и т.д.) кто-то другой агрессивно скрапит — вы получаете последствия.

## Как я это проверил {#comment-je-l-ai-verifie}
Я начал собирать статистику по ASN:

- процент успеха,
- количество 429,
- время деградации,
- восстановление.

Через несколько недель проявилась чёткая тенденция:

- как только ASN «срывается»,
- падают все IP внутри него,
- смена адреса не помогает (смена ASN — да).

Это не формальное доказательство, но корреляция была слишком стабильной для случайности.

<b>Важный момент:</b> часть падений производительности приходит от глобальных инцидентов вне локального контроля.
При масштабных сбоях Cloudflare несколько пулов могут падать параллельно, даже при хорошей гигиене скрапинга.

![Влияние глобального инцидента на статистику прокси](img/asn/proxy_stats.png)

> В масштабах интернета инциденты крупных операторов быстро становятся общими проблемами.

## Система «Светофор» {#le-systeme-feu-tricolore}

Чтобы автоматизировать контроль качества прокси, я внедрил систему «Светофор».
Принцип прост, как в логистике.

| Состояние | Условие | Действие |
| ---- | --------- | ------ |
| **Зелёный** | Адрес стабилен, успех выше порога | Активное использование в ротации |
| **Жёлтый** | Серия из 10 ошибок подряд | Пауза на несколько часов |
| **Красный** | Новый дрейф после фазы отдыха | Исключение из пула |

![Workflow «Светофор» для контроля ASN-магистралей](img/asn/feu_tricolor.png){: .shadow }

Этот workflow иллюстрирует непрерывную проверку «дороги» (ASN), а не только отдельного IP.

## Роль ASN в «Светофоре» {#role-de-l-asn-dans-le-feu-tricolore}

Со временем я добавил уровень агрегации:

Адрес → ASN → Пул.

Если несколько IP одного ASN начинают генерировать ошибки, приоритет всего диапазона снижается.

На практике система учится избегать проблемных магистралей.

Параллельно серверный job несколько раз в день проверяет новые сетевые маски и сравнивает их с внутренней историей.
Этот блок не предоставляется Webshare из коробки: это самодельный слой, улучшающий проактивный отбор пулов.

![Автоматическое обновление сетевых масок](img/asn/update_masks.png){: .shadow }

> Без пригодного endpoint'а даже очень хороший ИИ-ассистент не угадает реальный поток.
> Сначала нужно понять, как страница генерирует свой динамический гипертекст, чтобы автоматизировать чисто.
{: .prompt-info }

## Почему это полезно

Представим:

- у прокси 99 % успеха,
- потом он деградирует,
- из-за ротации система замечает это только через несколько дней.

За это время тысячи запросов уходят в пустоту — с реальной стоимостью.

«Светофор» позволяет отфильтровать плохие адреса за часы, а не за недели.

## Почему дорожная аналогия работает {#pourquoi-l-analogie-routiere-fonctionne}

ASN — это автомагистраль.

- IP — машины.
- Прокси — водители.
- Трафик — груз.

Машины можно менять, но если дорога забита — вы стоите.
Я перестал воевать с отдельными машинами и начал выбирать дороги.

Иначе вы просто сжигаете бюджет.

## Что вы можете сделать уже сейчас {#ce-que-vous-pouvez-faire-des-maintenant}

Если вы занимаетесь скрапингом:

1. Начните логировать ASN.
2. Сравнивайте процент успеха по диапазонам.
3. Отключайте проблемные сети блоками.
4. Автоматизируйте отбор.
5. Не доверяйте «дешёвым» пулам без статистики.

Это окупается очень быстро.

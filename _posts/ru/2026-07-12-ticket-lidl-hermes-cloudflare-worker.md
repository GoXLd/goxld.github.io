# От кассового чека к агенту: никогда не терять список покупок

Я хотел, чтобы мой агент мог предложить блюдо на основе того, что я реально купил, — без фотографирования чека и переписывания списка в Telegram. Источник уже существует: цифровые чеки моего аккаунта лояльности. Проблема не в том, чтобы показать их один раз. А в том, чтобы получать их без хрупкой браузерной сессии, превращать в чистые данные и гарантировать, что временный сбой агента не заставит покупку исчезнуть.

Поэтому я собрал небольшой сервис на Cloudflare Workers. Он выполняется **дважды в день**, читает максимум **10 последних чеков**, сохраняет новые в D1 и отдельно доставляет их содержимое Hermes. Первый коммит — это **13 файлов** и **744 добавленные строки**: достаточно мало, чтобы перечитать целиком, но с настоящей границей между сбором, персистентностью и уведомлением.

> Одной фразой: чек записывается в D1 до любой доставки; недоступный webhook создаёт задержку, а не потерю данных.
{: .prompt-info }

## Отправная точка

| Зона | Состояние до |
|---|---|
| Список покупок для агента | ручной ввод или отсутствие контекста |
| Цифровой чек | HTML, задуманный для отображения, а не стабильное бизнес-API |
| Аутентификация | не переигрывать логин и не управлять браузером в проде |
| Доставка агенту | невозможна напрямую от Telegram-бота к Telegram-боту |
| Сбой webhook'а | никогда не должен приводить к потере чека |

Первоначальной ловушкой был Telegram. Два бота не могут писать друг другу напрямую: сервис, читающий чеки, не может «отправить сообщение Hermes», как отправил бы человеку. Доставка агенту идёт через подписанный HTTP-webhook; Telegram остаётся для алертов, адресованных человеку.

## Метод: аутентифицировать, сохранить, затем доставить

Сайт и мобильное приложение программы лояльности опираются на одного провайдера идентичности. Worker хранит `refresh_token` как секрет и на каждом запуске обменивает его на короткоживущий `access_token`. Тот позволяет вызывать API чеков без хранения пароля, без CAPTCHA и без браузерной автоматизации.

Весь поток укладывается в простую идею: у каждого шага есть своё долговременное состояние.

```text
cron
  → refresh_token → access_token
  → список последних чеков
  → HTML-детали каждого неизвестного чека
  → парсинг позиций
  → INSERT в D1 с notified_at = NULL
  → POST на webhook Hermes
  → notified_at = отметка времени доставки
```

Основная таблица разделяет, в частности, идентификатор чека, его дату, сумму, JSON позиций, версию парсера и `notified_at`. Последняя колонка — это контракт восстановления: записанный, но не доставленный чек остаётся выбираемым при следующем проходе.

>Правила, не подлежащие обсуждению:
>1. **Персистентность до уведомления** — агент никогда не становится единственной копией покупки.
>2. **Помечать доставленным только после успеха** — ответ с ошибкой оставляет чек в ожидании.
>3. **Останавливать доставку на первой ошибке** — последующие чеки тоже сохраняют порядок и статус ожидания.
{: .prompt-danger }

## Извлекать данные из HTML, не доверяя HTML

Детали чека приходят в виде `htmlPrintedReceipt`. Видимый текст похож на моноширинный чек: удобно для человека, плохой контракт для парсера. К счастью, каждая строка позиции несёт атрибуты `data-*` с идентификатором, ценой за единицу и количеством.

```js
// Атрибуты дают числа; видимый текст даёт читаемое имя.
const name = visible.split(/\s{2,}/u)[0].trim();
const unitPrice = parseDecimal(attr(tag, 'data-unit-price'));
const quantity = parseDecimal(attr(tag, 'data-art-quantity')) ?? 1;
const lineTotal = parseDecimal(decimals.at(-1));
```

Важная деталь — имя. Атрибут описания товара иногда содержит криво закодированные байты; поэтому парсер берёт название из видимого текста с декодированными HTML-сущностями. Затем он сверяет сумму со строкой «À payer» чека.

| Элемент | Выбранный источник | Причина |
|---|---|---|
| Количество | атрибут `data-art-quantity` | структурированное значение |
| Цена за единицу | атрибут `data-unit-price` | структурированное значение |
| Название товара | декодированный видимый текст | избегает битых символов |
| Сумма | строка «À payer» | проверка согласованности чека |

Тест парсинга опирается на реальный анонимизированный чек: он проверяет, что в нём минимум **30 позиций**, что каждая строка имеет имя и числовую сумму, что упаковка сохраняет своё количество и что название с диакритикой корректно декодируется. Второй тест требует возврата `null` для страницы, не являющейся чеком.

## Кусок, который важен: переигрываемая доставка

В Worker'е сбор и доставка — две отдельные петли. Первая вставляет только неизвестные чеки. Вторая запрашивает у D1 все записи с `notified_at = NULL`, постит их по одной на webhook и помечает только после успешного вызова.

```js
for (const row of pendingReceipts) {
  const ok = await notifyHermes(env, row);
  if (!ok) break; // чек остаётся в ожидании до следующего прогона
  await markNotified(env, row.id);
}
```

Это намеренно проще полноценной очереди, но поведение прозрачно: webhook отсутствует, таймаут или агент временно недоступен — следующий cron продолжит с того места, где остановился предыдущий. Сервис не объявляет воображаемый успех и не обновляет `notified_at` из оптимизма.

## Честная часть: ложные пути

Первым рефлексом было бы использовать Telegram везде. Он по своей конструкции не годится для обмена бот-бот. Webhook менее наглядный, но это правильный канал машина-машина; бот уведомлений остаётся полезным для того, чтобы предупредить человека об истёкшей аутентификации.

Второе ограничение: парсер по-прежнему зависит от HTML-структуры и атрибутов, которые провайдер отдаёт сегодня. Я изолировал этот риск в `parseReceiptHtml()` и записываю `parser_version` с каждым чеком. Если HTML изменится, я смогу поправить парсер и знать, какой версией произведена каждая запись, вместо того чтобы размазывать регулярку по всему Worker'у.

Наконец, `refresh_token` не вечен. Если его инвалидируют, выполнение алертит администратора и требует переиграть OAuth-поток. Это не тихий сбой, но и не магическая автономия.

## Итоги

| Метрика | Результат |
|---|---|
| Частота синхронизации | **2 раза в день** |
| Окно чтения | **10 последних** чеков |
| Доставка | подписанный HTTP-webhook, переигрываемая |
| Персистентность | D1 до уведомления |
| Тесты парсинга | реальный анонимизированный чек + некорректный HTML |
| Точка восстановления | `notified_at = NULL` |

Результат — не «ИИ, читающий мои покупки». Это скромный пайплайн, дающий агенту контекст с известным происхождением: чек, структурированные позиции, дата и статус доставки.

Правильный вопрос теперь не «как отправить список боту?», а «какое доказательство мы храним, когда промежуточная система падает?». В этом случае колонка `notified_at` и порядок «персистентность → доставка» дают надёжности больше, чем любая эффектная интеграция. Hermes может рассуждать о последней покупке, а Worker сохраняет более приземлённую ответственность — ничего не забыть.

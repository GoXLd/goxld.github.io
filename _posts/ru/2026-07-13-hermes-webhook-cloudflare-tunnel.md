# Hermes за туннелем: не путать доступность и доставку

Я хотел передавать Hermes покупки, уже структурированные удалённым Worker'ом. Первая идея кажется банальной: выставить HTTP-URL, сделать `POST` и ждать подтверждения. На практике агент работает на машине, которая не должна становиться публичным сервером, а отправитель и получатель могут падать независимо друг от друга.

Ловушка была и в канале: Telegram-бот не может быть надёжным транспортом к другому боту. Поэтому я разделил роли: Worker порождает событие, подписанный webhook доставляет его Hermes, а Telegram остаётся каналом для человека.

> Одной фразой: Cloudflare Tunnel делает Hermes достижимым без открытия личного компьютера; HMAC-подпись решает, может ли запрос войти, а долговременное состояние на стороне Worker'а решает, нужно ли его переигрывать.
{: .prompt-info }

## Отправная точка

| Зона | Исходное состояние |
|---|---|
| Агент Hermes | Сервис доступен локально, не публичное API для прямого выставления |
| Производитель событий | Cloudflare Worker, умеющий ходить по HTTP |
| Изначально задуманный транспорт | Telegram-уведомление между двумя ботами |
| Основной риск | Потерянный или подделанный запрос нужно было отличать от успеха |
| Доступная проверка | Тестовый маршрут, отвечающий `202 Accepted`, когда webhook здоров |

Именованный туннель работает на хосте, где выполняется Hermes — не на моём Mac. Он проксирует только маршрут webhook'а к локальному сервису. Это важное различие: я не открывал входящий порт на личной машине и не выкладывал весь интерфейс Hermes в интернет.

## Минимальная граница: туннель, маршрут, подпись

Потоку не нужно быть сложным, чтобы быть проверяемым: производитель готовит тело, считает подпись с общим секретом и вызывает публичный URL туннеля. Hermes проверяет подпись до обработки события.

```text
Cloudflare Worker
  → POST, подписанный HMAC
  → Cloudflare Tunnel
  → маршрут /webhooks/… на хосте Hermes
  → проверка подписи
  → обработка события
  → 202 Accepted
```

>Правила, принятые на границе:
>1. **Секрет никогда не кладётся в URL** — тело и подпись путешествуют раздельно.
>2. **Туннель публикует только полезный маршрут** — это не лазейка для выставления локального интерфейса агента.
>3. **`202` не является доказательством персистентности на стороне производителя** — тот хранит собственное состояние для повторов.
{: .prompt-danger }

Тест webhook'а использует тот же маршрут, что и Worker, и действительно вернул `202 Accepted`. Это проверка доступности: туннель доходит до сервиса, и маршрут принимает аутентифицированный запрос. Это сознательно не полный тест бизнес-содержимого.

## Почему туннель не заменяет очередь

Туннель отвечает на один вопрос: *как достучаться до локального сервиса, не выставляя его напрямую?* Он не отвечает на следующий: *что делать, если Hermes на мгновение недоступен после сбора события?*

Ответ остаётся на стороне Worker'а: событие записывается до попытки доставки, а его статус обновляется только после успешного ответа. В текущем пайплайне **10 тикетов** первой страницы были доставлены Hermes; эта цифра подтверждает первый проход, а не магическую гарантию для всех будущих инцидентов.

| Ситуация | Что решает туннель | Что должен решать производитель |
|---|---|---|
| Hermes доступен локально | HTTPS-URL без открытого порта на личной машине | Ничего больше |
| Запрос без валидной подписи | Ничего: он доходит до границы | Hermes его отклоняет |
| Туннель или сервис недоступны | Ничего: вызов падает | Сохранить событие и повторить |
| Ответ `202 Accepted` | Маршрут принял доставку | Пометить событие доставленным только по определённому контракту |

Это разделение предотвращает частую путаницу: достижимый endpoint — это не очередь. Туннель переносит попытку; персистентность и статус доставки делают попытку переигрываемой.

## Честная часть: ещё одна сетевая поверхность

Туннель избавляет от открытия входящего порта, но всё же создаёт публичный URL. Ошибка конфигурации может расширить экспонируемую поверхность далеко за пределы задуманного webhook'а. Я сохранил выделенный маршрут и HMAC-подпись именно для того, чтобы «знать URL» было недостаточно для инъекции события.

Другое ограничение: тест `202` не доказывает ни качество парсинга на стороне Worker'а, ни финальный эффект в Hermes. Он проверяет сантехнику. Тесты данных и механизм повторов остаются отдельными обязанностями; их смешение дало бы ложное ощущение надёжности.

## Итоги

| Элемент | Результат |
|---|---|
| Сетевая экспозиция | Cloudflare Tunnel к выделенному маршруту webhook'а |
| Защита на входе | HMAC-подпись, без секрета в URL |
| Проверка здоровья | `202 Accepted` |
| Хост туннеля | Машина Hermes, не личный компьютер |
| Первая наблюдаемая доставка | **10 тикетов** первой страницы |
| Гарантия повторов | Персистентность и статус на стороне Worker'а, не туннель |

Результат — это не «Hermes, открытый в интернет». Это узкая граница между удалённым производителем и локальным агентом: достаточно доступная, чтобы принять событие, и достаточно контролируемая, чтобы отклонить запрос без правильной подписи.

Полезный вопрос теперь не «как достучаться до агента из Worker'а?», а «какая часть системы доказывает, что событие принято, обработано и при необходимости переиграно?». Туннель чисто отвечает на первую часть; остальное должно оставаться явным в контракте доставки.
